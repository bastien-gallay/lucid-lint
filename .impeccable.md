# .impeccable.md

Design context for `lucid-lint`. This file is the source of truth for all
frontend and brand work on the project — website, mdBook docs, any future
editor integrations, social cards, diagrams.

## Design Context

### Users

Three audiences share the docs site and the eventual marketing surface:

1. **Evaluators** — tech writers, docs maintainers, developer-relations, and
   staff engineers scanning the site for 90 seconds to decide whether the
   tool belongs in their CI. They want to understand the stance, see a
   credible example, and leave.
2. **Contributors** — people writing or proposing rules, word lists, or
   translations. They need RULES.md, AGENTS.md, and the architecture
   pages to be navigable and to reward patience.
3. **Cognitive-accessibility community** — ADHD, dyslexic, low-attention,
   fatigued, L2, and FALC/RGAA/EAA practitioners auditing whether the
   project walks its talk. For this audience the *docs UI itself* is a
   credibility test. If it does not feel good to them, the pitch fails
   regardless of the rules shipped.

**Context of use**: evaluation in a browser tab alongside four others;
long-form reading on contribution days; occasional late-night reading
(dark mode path must be first-class, not a grudging toggle).

### Brand personality

Four words: **reassuring, professional, innovative, gentle**.

Expanded:

- **Reassuring**: no shouting, no anxiety, no urgency tricks. The tool
  finds problems in prose — that is already a vulnerable genre. The UI
  should lower the stakes of a finding, not raise them.
- **Professional**: it is real software with real research behind it
  (Sweller, Gibson, Graesser, Coh-Metrix). No toy energy, no emojis-in-
  headings. Precision in language, typography, and spacing.
- **Innovative**: the category is sleepy (Vale, textlint, Hemingway).
  This project takes a distinct stance — bilingual-first, cognitive-load-
  first. The design should signal that stance before the reader clicks.
- **Gentle**: palette, type, and motion are on the calm side of every
  decision. No glow, no bounce, no "AI accent."

**Emotion the interface should evoke**: *"oh — someone finally made this
for me."* Not awe, not impressed. Recognition and relief.

### Reference sites (and what specifically)

- **[impeccable.style](https://impeccable.style)** — clarity. Every
  element shows literally what it is. No stock-photo gesturing.
  Steal: directness, absence of decorative placeholders, trust in content.
- **[haskell.org](https://haskell.org)** — welcoming entry to a hard
  subject. Makes nerds feel seen *and* makes newcomers feel invited.
  Steal: the "technical but not gatekept" balance, generous headings,
  example-driven above the fold.
- **[pi.dev](https://pi.dev)** — immersive minimalism. Shows what the
  product does above a single horizon line, nothing more.
  Steal: ruthless above-the-fold, one demonstration, no feature grid.
- **[claude.com](https://claude.com)** — neutral warm palette (terracotta
  family), high readability, restrained accents.
  Steal: the warm near-neutral dominant with a single confident accent.

### Anti-references

Do **not** look like:

- Institutional or government-portal design (overly corporate, cold,
  compliance-theatre). This project serves the FALC/EAA audience but is
  not a ministry website.
- Medical software. No pale blue + white + iconography of clipboards.
- "Yet another dashboard widget." No SaaS hero with a product screenshot
  slightly tilted in a gradient backdrop.
- **Sonar / code-quality enterprise tools.** The project is not a quality
  gate; it is a reading companion.
- Educational / edtech platforms. Not here to teach the user to write;
  here to surface friction.
- Every other Rust crate docs site (`docs.rs` sameness, stock mdBook
  `rust` + `navy`, default Open Sans + Source Code Pro).

### Aesthetic direction

- **Theme**: light and dark as equals. Both must feel like the *canonical*
  version of the site. The "reassuring" target sits between them —
  neither clinical-white nor late-night-hacker-dark. Expect roughly 50/50
  split among readers.
- **Palette** (proposed by Bastien, WCAG-checked):

  | Token | Value | Role | Contrast on Paper |
  |---|---|---|---|
  | `--color-paper` | `#FAFAF7` | default light background (warm off-white) | — |
  | `--color-ink` | `#1A1D24` | primary text (warm near-black with slight blue) | 15.8:1 ✅ AAA |
  | `--color-lucid` | `#2E5EAA` | primary accent / links (confident muted blue) | 7.2:1 ✅ AAA |
  | `--color-aurora` | `#8B5FBF` | secondary accent (violet) — sparingly | 5.1:1 ✅ AA |
  | `--color-fog` | `#6B7280` | secondary text / borders | 4.9:1 ✅ AA (reserve for secondary text only) |

  Convert all tokens to **OKLCH** in CSS (keep hex references in this
  document as source of truth). Dark mode is derived by inverting
  Paper/Ink and re-checking contrast — not by tossing in a saturated
  cyan accent.

- **Typography direction** (not final — the `/typeset` pass will commit):
  - **Body**: reading-optimized sans — **Atkinson Hyperlegible Next**
    (Braille Institute, open license) is the strong-signal default for
    this audience. It directly demonstrates the project's stance.
  - **Display**: a gentle, bookish, slightly humanist partner —
    **Literata** (designed by TypeTogether for Google Play Books) or
    **Faustina** are candidates. Both are reading-optimized, both are
    outside the "reflex font" pool.
  - **Mono**: **Recursive Mono** (variable, distinctive) or
    **Commit Mono**. Not IBM Plex, not JetBrains Mono, not Source Code Pro.
  - **Opt-in toggle**: header control offering `Standard / Hyperlegible /
    OpenDyslexic`, persisted in `localStorage`. OpenDyslexic is included
    for readers who specifically prefer it, but Atkinson is the default
    because it is more universally legible and less stigmatizing.

- **Logo**: two-part mark designed by Bastien — a handwritten/messy wave
  entering from the left, passing through a **lens** (circle + focal
  dot), exiting as a **clean horizontal line**. Literal metaphor for the
  product. Two variants:
  - **Primary**: solid `--color-lucid` ring → `--color-ink` line
  - **Secondary**: `--color-lucid` → `--color-aurora` gradient on the
    ring stroke, solid ink on the line. (Gradient on SVG stroke is
    allowed; the ban is specifically on gradient *text fill*.)

  The lens motif can extend into: favicons, section dividers (the clean
  line after the lens), empty states (the messy-wave-only, pre-lens), a
  reading-progress indicator on long pages.

### Design principles

Five principles that govern every design decision on this project:

1. **Dogfood the audience.** The docs site is the product's first proof.
   If it fails ADHD/dys* readers, the pitch is broken. Generous leading
   (≥ 1.65), ≤ 75ch line length, a visible reading-preferences widget,
   AAA contrast, `prefers-reduced-motion` respected absolutely.
2. **Literal over decorative.** Like `impeccable.style` and `pi.dev`,
   every visual element should show what the tool does — not gesture at
   it. The logo is literal. The empty states should be literal. No
   decorative sparklines, no illustrative icons that mean nothing.
3. **Reassuring restraint.** When in doubt, calmer, quieter, slower.
   Professional without being institutional; warm without being cute.
   No glow, no bounce, no "AI accent." Motion is used for state, not
   delight theatre.
4. **Bilingual equality from day one.** EN and FR are peers, not
   primary-and-translated. `lang` attribute handled correctly, both
   languages proof-read in the same revision, no "French coming soon"
   notice.
5. **Dogfood the linter.** Prose on the site must pass `lucid-lint`'s
   own rules at the `public` profile. Add `just dogfood-docs` to CI so
   the docs can never regress below the bar the tool sets for its
   users.

### Accessibility target

**WCAG 2.2 Level AAA** is the stated bar — this is an accessibility tool
and the ceiling is the only coherent choice. Practically:

- Normal body text: contrast ≥ 7:1.
- Large text / UI: contrast ≥ 4.5:1.
- Interactive targets: minimum 44×44 px (iOS HIG), focus ring visible
  without relying on color alone.
- Respect `prefers-reduced-motion: reduce` — no decorative animation,
  no parallax, no auto-playing anything.
- Keyboard: every interactive surface reachable, focus order logical,
  skip-to-content link present.
- Reading preferences persisted in `localStorage` and respected across
  visits.
- Language-switch control visible in the header (EN / FR), aria-labelled.

Where AAA is genuinely impractical (e.g., some UI-chrome contrast on
third-party embeds), document the exception in this file rather than
silently dropping the bar.

## Notes for future design work

- The `/critique` command should be rerun against the rendered site
  (`just docs-serve`, then the `chrome-devtools` MCP) after any
  significant design change.
- The `/typeset` command will make the binding font decision. Until
  then, treat the typography section above as a *shortlist*, not a
  commitment.
- Before proposing any new visual element, ask: "does this help an ADHD
  reader understand the site in the next 8 seconds?" If not, cut it.
